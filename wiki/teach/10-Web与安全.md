# 第 10 章：Web 与安全

前两章我们了解了服务器基础和 Python 部署工具，这一章我们来看看 Web 相关的技术：Nginx、DNS、HTTPS 和 SSL 证书。

---

## 一、Nginx 是什么？

Nginx（读作 "engine X"）是一个高性能的 Web 服务器软件。它的作用是监听来自互联网的请求，然后把相应的内容返回给访问者。

除了作为 Web 服务器，Nginx 还经常被用作反向代理、负载均衡器、邮件代理等等。在这一章里，我们主要关注它作为 Web 服务器和反向代理的用法。

### Nginx 作为 Web 服务器

作为 Web 服务器，Nginx 的工作很简单：当有人访问你的网站时，Nginx 找到对应的文件，然后把文件内容返回给访问者。

比如，访问 `nanoshiki.top/index.html` 时，Nginx 去网站根目录找到 index.html 文件，把它的内容返回给浏览器，浏览器就显示出网页了。

Nginx 非常适合提供静态文件（HTML、CSS、JavaScript、图片等等），它的性能很高，能同时处理大量并发连接。

### Nginx 作为反向代理

除了作为 Web 服务器，Nginx 还经常被用作反向代理。反向代理是什么意思呢？

想象一下，你有多个服务运行在同一个服务器上，有的在 5000 端口（Flask 后端），有的在其他端口。你不想让访问者在网址里加上端口号，也不想把这些端口直接暴露在互联网上。

这时候反向代理就派上用场了。Nginx 监听 80（HTTP）和 443（HTTPS）端口，所有的请求都先到 Nginx 这里。Nginx 根据请求的网址，把请求转发给后端相应的服务。

比如：
- 访问 `nanoshiki.top/` 时，Nginx 返回静态网页
- 访问 `nanoshiki.top/api/` 时，Nginx 把请求转发给运行在 5000 端口的 Flask 后端

这样，访问者只需要访问 80 或 443 端口，不需要知道后端服务运行在哪个端口，后端服务的端口也不需要对外公开，更安全。

### Nginx 的配置文件

Nginx 的配置文件通常在 `/etc/nginx/` 目录下。不同的 Linux 发行版，配置文件的组织方式略有不同。

在 Ubuntu 和 Debian 系统上，通常有两个目录：`sites-available` 和 `sites-enabled`。你在 `sites-available` 里创建配置文件，然后在 `sites-enabled` 里创建一个软链接指向它，Nginx 就会加载这个配置了。

在 CentOS、RHEL 和 OpenCloud 系统上，通常只有一个 `conf.d` 目录。你直接把配置文件放在 `conf.d` 里，Nginx 就会自动加载所有 `.conf` 结尾的文件。

配置文件的语法并不复杂，它由一个个 `server` 块组成，每个 `server` 块对应一个网站或一个域名的配置。

在 `server` 块里，你可以指定监听的端口、域名、根目录、以及各种路径的处理规则。比如，`location /` 块处理根路径的请求，`location /api/` 块处理 API 请求。

这里是一个简单的配置示例：

```nginx
server {
    listen 80;
    server_name nanoshiki.top www.nanoshiki.top;

    location / {
        root /var/www/myWeb;
        index index.html;
        try_files $uri $uri/ =404;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

这个配置的意思是：
- 监听 80 端口
- 处理 nanoshiki.top 和 www.nanoshiki.top 的请求
- 访问根路径 / 时，从 /var/www/myWeb 目录提供文件
- 访问 /api/ 时，转发给 http://127.0.0.1:5000

## 二、什么是 DNS？

想象一下，如果我们想要去朋友家做客，我们需要知道朋友家的地址。同样的，当我们在浏览器里输入一个网址时，浏览器也需要知道这个网站所在的服务器的地址。

服务器的地址是一串数字，叫做 IP 地址，比如 `106.54.203.176`。但是让人们记住这一串数字实在是太难了，所以我们发明了域名，比如 `nanoshiki.top`。

DNS，也就是域名系统（Domain Name System），就像是互联网的电话簿。它的工作很简单：把我们容易记住的域名，转换成电脑能理解的 IP 地址。

### DNS 的工作原理

当你在浏览器里输入 `nanoshiki.top` 时，浏览器会先去问 DNS 服务器："嘿，nanoshiki.top 对应的 IP 地址是什么呀？" DNS 服务器查找到后，把 IP 地址告诉浏览器，浏览器就知道该去哪里找这个网站了。

这个过程看起来简单，但实际上背后有一套复杂的系统。DNS 服务器是分层的：

- 首先问本地 DNS 服务器（通常是你的 ISP 提供的）
- 如果本地 DNS 不知道，再问根域名服务器
- 根域名服务器告诉去问顶级域名服务器（比如 .top 的服务器）
- 顶级域名服务器告诉去问权威域名服务器（也就是 nanoshiki.top 的 DNS 服务器）
- 最后拿到 IP 地址

好消息是，这些都自动发生，我们不需要关心细节，只需要在域名注册商那里设置好 DNS 记录就行了。

### DNS 记录类型

DNS 不只是能把域名转换成 IP 地址，它还能记录很多其他信息。最常见的几种记录类型是：

- **A 记录**：把域名指向一个 IPv4 地址，这是我们最常用的。
- **AAAA 记录**：把域名指向一个 IPv6 地址，是未来的趋势。
- **CNAME 记录**：把一个域名指向另一个域名，有点像别名。
- **TXT 记录**：可以存储任意文本信息，我们后面会讲到，SSL 证书验证时会用到它。
- **MX 记录**：指定邮件服务器，告诉别人发邮件到这个域名时该送到哪里。

对于网站部署，最常用的是 A 记录，把域名指向服务器的公网 IP。

### DNS 传播延迟

当你在域名注册商那里修改了 DNS 记录后，并不是全世界立刻都知道这个变化。因为 DNS 服务器遍布全球，它们之间需要时间来同步这些信息。

这个同步的过程就叫做 DNS 传播，通常需要几分钟到几小时，有时候甚至需要 24 小时。这就是为什么有时候你明明已经改了 DNS 记录，但有些地方还是访问不到新地址的原因。

你可以用 `nslookup` 或 `dig` 命令查看不同 DNS 服务器看到的记录：

```bash
# 用 8.8.8.8（Google DNS）查询
nslookup nanoshiki.top 8.8.8.8

# 用 1.1.1.1（Cloudflare DNS）查询
nslookup nanoshiki.top 1.1.1.1
```

如果这些显示的还是旧 IP，说明 DNS 还没传播完成，只能耐心等待。

## 三、为什么需要 HTTPS？

在以前，网站大多使用 HTTP 协议来传输数据。HTTP 协议的问题在于，它传输的数据都是明文的，就像一张明信片，任何人在传输过程中都能看到里面的内容。

如果你只是浏览一些公开的新闻网站，这可能没什么大问题。但如果你在输入密码、银行账号，或者在网购时输入信用卡信息，明文传输就太危险了。想象一下，你的密码在传输过程中被别人看到了，那会是什么后果。

HTTPS 的出现就是为了解决这个问题。HTTPS 在 HTTP 的基础上加了一层加密，让传输的数据变成了只有发送方和接收方才能看懂的密文。这就像是把明信片放进了一个密封的信封里，只有收信人才能打开看到内容。

### HTTPS 的工作原理

HTTPS 使用了一种叫做非对称加密的技术。简单来说，它有两把钥匙：一把公钥，一把私钥。

公钥是公开的，任何人都可以拿到。私钥是保密的，只有服务器自己知道。

当你访问一个 HTTPS 网站时，服务器会把它的公钥发给你。你的浏览器生成一个随机的密钥，用服务器的公钥加密后发给服务器。服务器用自己的私钥解密，拿到这个随机密钥。

接下来，双方就用这个随机密钥来加密通信。因为只有你们两个知道这个随机密钥，所以别人就算截获了数据也看不懂。

这整个过程听起来有点复杂，但好消息是，这些都是浏览器和服务器自动完成的，我们作为用户，只需要看到浏览器地址栏里那个小锁图标，就知道通信是安全的了。

## 四、SSL 证书是什么？

虽然 HTTPS 加密了通信，但还有一个问题：你怎么知道你正在通信的服务器真的是你想访问的那个网站，而不是别人冒充的呢？

这就需要 SSL 证书了。SSL 证书就像是网站的身份证，它由一个受信任的第三方机构颁发，这个机构叫做证书颁发机构，简称 CA。

### SSL 证书的作用

SSL 证书里包含了网站的域名、公钥，以及 CA 的签名。当你访问 HTTPS 网站时，服务器会把 SSL 证书发给你。你的浏览器会检查这个证书：

- 它是不是由受信任的 CA 颁发的？
- 证书里的域名和你正在访问的域名一致吗？
- 证书过期了吗？

如果一切检查都通过，浏览器就会显示那个小锁图标，告诉我们连接是安全的。如果证书有问题，浏览器会发出警告，阻止我们继续访问，或者至少提醒我们小心。

这就像是你去银行办事，银行职员会先看你的身份证，确认你是你本人，才会给你办理业务。SSL 证书就是网站的身份证。

### Let's Encrypt：免费的 SSL 证书

在以前，申请 SSL 证书是需要花钱的，而且价格不便宜。这让很多小型网站和个人网站难以承担。

后来出现了 Let's Encrypt，这是一个非营利的证书颁发机构，它提供免费的 SSL 证书。任何人都可以申请，而且申请过程很简单，很快就能拿到证书。

Let's Encrypt 的证书有效期是 90 天，不过我们可以用工具自动续期，所以不用担心过期的问题。

### Certbot：自动申请和续期证书

Certbot 是 Let's Encrypt 官方推荐的工具，它可以帮助我们自动申请、安装和续期 SSL 证书。

Certbot 支持很多种 Web 服务器，比如 Nginx、Apache 等等。它会自动修改服务器的配置文件，把证书配置好，我们几乎不需要手动操作什么。

Certbot 申请证书时有几种验证方式，最常见的两种是：

- **HTTP 验证**：Certbot 在你的网站根目录下创建一个临时文件，然后 Let's Encrypt 尝试访问这个文件。如果能访问到，就证明你确实拥有这个域名。
- **DNS 验证**：Certbot 让你在 DNS 里添加一条 TXT 记录。Let's Encrypt 查询这条记录，如果存在且内容正确，就证明你拥有这个域名。

HTTP 验证最简单，但它要求你的网站已经可以通过 HTTP 访问，而且 DNS 已经传播完成。如果 DNS 还在传播中，或者你的网站还没配置好，DNS 验证会是更可靠的选择。

## 五、把这些知识串起来

现在我们已经了解了服务器基础、Python 部署工具、以及 Web 与安全相关的技术，让我们看看它们是如何一起工作，让你的网站在互联网上安全访问的。

整个流程大概是这样的：

1. 你在域名注册商那里买了一个域名 `nanoshiki.top`，然后添加了一条 A 记录，把它指向你的服务器 IP 地址 `106.54.203.176`。

2. DNS 记录需要时间传播到全球各地的 DNS 服务器。

3. 你在服务器上安装了 Nginx，配置好网站的根目录和反向代理规则。

4. 你使用 venv 创建了 Python 虚拟环境，安装了 Flask 和 Gunicorn。

5. 你用 Systemd 创建了服务，让 Gunicorn 自动启动、自动重启。

6. 你使用 Certbot 申请 SSL 证书。Certbot 验证你确实拥有这个域名（通过 HTTP 或 DNS 验证），然后从 Let's Encrypt 那里拿到证书。

7. Certbot 自动修改 Nginx 配置，把证书配置好，并且把 HTTP 请求重定向到 HTTPS。

8. 现在，当一个访问者在浏览器里输入 `nanoshiki.top` 时：
   - 浏览器问 DNS 服务器 `nanoshiki.top` 的 IP 是什么
   - DNS 服务器返回 `106.54.203.176`
   - 浏览器连接到这个 IP 的 443 端口
   - Nginx 收到请求，把 SSL 证书发给浏览器
   - 浏览器验证证书，确认无误
   - 浏览器和服务器建立加密连接
   - Nginx 返回网页内容，或者把请求转发给后端服务
   - 浏览器显示出你的网站，地址栏有一个小锁图标

整个过程听起来步骤很多，但大部分都是自动发生的，我们只需要做好初始的配置就可以了。

## 六、常见问题

### DNS 还没传播怎么办？

如果你刚修改了 DNS 记录，但 Let's Encrypt 还在使用旧的 IP，不要着急。你可以：

1. 等待一段时间，几小时到 24 小时都有可能。
2. 使用 DNS 验证方式申请证书，这种方式不需要网站可以访问，只需要你能添加 DNS 记录。

### SSL 证书申请失败怎么办？

如果 Certbot 报错说验证失败，先看看错误信息：

- 如果是 404 错误，说明 Let's Encrypt 访问验证文件时失败了，检查 Nginx 配置里是否正确处理了 `.well-known` 路径。
- 如果是 403 错误，说明权限有问题，检查验证文件所在目录的权限是否正确。
- 如果提示域名解析到了错误的 IP，那就是 DNS 传播的问题，参考上一个问题的解决办法。

### 证书过期了怎么办？

如果你使用 Certbot，不用担心，它会自动设置一个定时任务，在证书过期前自动续期。你也可以手动运行 `certbot renew` 来测试续期是否正常工作。
